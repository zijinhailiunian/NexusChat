# 需求到任务拆解流程（迭代节奏）

目标：保证持续推进，不跑偏；每个任务都有明确验收条件和完成证据。

## 1) 需求澄清（输入 / 输出 / 边界）

在开始开发前，必须先回答以下问题：

- 输入是什么？（用户故事、接口入参、触发条件）
- 输出是什么？（页面行为、接口响应、日志产物）
- 边界是什么？（不做什么、异常场景、性能/安全约束）

> 建议在 Issue/PR 中固定使用「输入 / 输出 / 边界」三个小节。

## 2) 拆任务（每项 0.5 ~ 2 天）

拆分规则：

- 每个任务应可独立提交并可评审。
- 单任务复杂度控制在 0.5~2 天内。
- 避免“超大 PR”；如超出范围，继续拆子任务。

推荐模板：

- Task A：
  - 范围：
  - 预估：
  - 依赖：

## 3) 定义验收标准（可测试）

每个任务必须有可执行验收标准，至少包括：

- 功能行为可验证（接口/页面/日志）。
- 自动化检查可执行（如 `npm test`、`npm run check`）。
- 文档同步更新（`README.md` / `docs/*`）。

建议格式：

- Given / When / Then
- 或 输入 -> 输出 的断言形式

## 4) 开发 + 自测

开发过程中，至少执行：

- `npm run format:check`
- `npm run lint`
- `npm test`
- `npm run check`

并在 PR 中附上关键输出作为证据。

## 5) PR 评审 + 合并

PR 必须包含：

- 需求澄清（输入 / 输出 / 边界）
- 任务拆解与范围说明
- 验收标准 + 验收结果
- DoD 5 项勾选

代码评审通过后方可合并。

## 6) 发布记录（Changelog）

每次合并后更新 `CHANGELOG.md`：

- Added / Changed / Fixed
- 关联 PR / 任务编号
- 对外可见行为变化与兼容性说明

---

## 完成证据要求（必须）

每个任务都必须可定位到以下证据：

1. 实现证据：提交与核心文件路径
2. 测试证据：命令与通过输出
3. 文档证据：更新的文档文件
4. 评审证据：PR 审批记录
5. 发布证据：`CHANGELOG.md` 条目
